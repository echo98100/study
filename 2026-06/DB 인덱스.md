### 인덱스가 필요한 이유

테이블에 데이터가 많아질수록 원하는 데이터를 찾기 위해서 테이블 전체를 스캔해야 하는데, 이를 **Full Table Scan**이라고 합니다.

데이터가 100만 건이라면 1건을 찾기 위해서 100만 번 비교해야 하는 상황이 생길 수 있는데요, 이를 해결하기 위한 것이 인덱스입니다.

인덱스는 특정 컬럼에 대한 정렬된 별도 자료구조를 만들어두고, 이를 통해 원하는 데이터의 위치를 빠르게 찾을 수 있습니다.

---

### 인덱스 내부 구조

대부분의 RDBMS는 **B+Tree**(Balanced+Tree) 구조로 인덱스를 구현합니다.

B+Tree의 핵심 특징은 다음과 같습니다.

- 항상 균형 트리를 유지합니다. 따라서 어떤 데이터를 찾아도 탐색 깊이가 동일해 O(log N)의 탐색 성능을 보장합니다.
- 리프 노드만 실제 데이터 주소(포인터)를 가집니다. 내부 노드는 경로 안내의 역할만 합니다.
- 리프 노드끼리 연결 리스트로 연결되어 있어, 범위 검색(BETWEEN, >, <)이 효율적입니다.

---

### 클러스터형 인덱스 vs 보조 인덱스

- **클러스터형 인텍스** (Clustered Index)

클러스터형 인덱스는 테이블 데이터 자체가 인덱스 순서대로 물리적으로 정렬되어 저장됩니다.

MySQL에서는 PK가 자동으로 클러스터형 인덱스가 되는데, 테이블당 하나만 존재합니다.

리프 노드에 실제 행 데이터 전체가 저장되기 때문에 PK로 조회하면 가장 빠릅니다.

- **보조 인덱스** (Secondary Index)

PK 외의 컬럼에 생성하는 인덱스입니다. 보조인덱스는 테이블 당 여러 개 생성 가능합니다.

리프 노드에는 실제 데이터가 아닌 해당 행의 PK 값이 저장됩니다.

그렇기 때문에 보조인덱스로 조회를 할 경우

보조 인덱스 탐색 → PK값 획득 → 클러스터형 인덱스 재탐색 → 실제 데이터 획득

의 과정을 거치고, 이를 북마크 룩업이라고 합니다.

### 커버링 인덱스

쿼리에서 필요한 모든 컬럼이 인덱스 안에 포함되어 있는 경우, 클러스터형 인덱스를 다시 조회하지 않고 인덱스만으로 결과를 반환할 수 있게됩니다.

```jsx
-- name, email 컬럼으로 복합 인덱스를 생성한 경우
CREATE INDEX idx_name_email ON users (name, email);

-- 이 쿼리는 인덱스만으로 결과 반환 가능 (커버링 인덱스)
SELECT name, email FROM users WHERE name = '관영';

-- 이 쿼리는 인덱스 탐색 후 PK로 재조회 필요 (커버링 인덱스 아님)
SELECT * FROM users WHERE name = '관영';
```

이때는 북마크 룩업의 과정이 생략되기 때문에 성능이 크게 향상됩니다.

### 복합 인덱스와 인덱스 선두 컬럼 원칙

위의 예시처럼 여러 컬럼을 묶어 하나의 인덱스로 만드는 것이 복합 인덱스입니다.

복합 인덱스에서 가장 중요한 원칙은 인덱스 선두 컬럼입니다.

복합 인덱스는 선두 컬럼부터 순서대로만 인덱스를 탈 수 있습니다.

```jsx
-- 인덱스 사용 O (선두 컬럼 name 포함)
WHERE name = '관영'
WHERE name = '관영' AND age = 25

-- 인덱스 사용 X (선두 컬럼 name 없음)
WHERE age = 25
```

선두 컬럼을 건너뛰면 인덱스를 전혀 활용하지 못하고 Full Table Scan이 발생하게 되기 때문에 복합 인덱스 설계 시 카디널리티가 높은 컬럼을 선두에 두는 것이 유리합니다.

---

### 인덱스를 타지 않는 경우

인덱스가 있더라도 활용되지 않는 경우가 있습니다.

1. **컬럼에 함수나 연산을 적용한 경우**

```jsx
-- 인덱스 사용 X
WHERE YEAR(created_at) = 2024
WHERE age + 1 = 26

-- 인덱스 사용 O
WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'
WHERE age = 25
```

1. **LIKE 패턴이 앞에 있는 경우**

```jsx
-- 인덱스 사용 X (앞에서부터 와일드카드)
WHERE name LIKE '%관영'

-- 인덱스 사용 O (뒤에서만 와일드카드)
WHERE name LIKE '관영%'
```

1. **NOT IN / NOT EXISTS / ≠ 사용 시**

부정 조건이 있을 경우 인덱스를 효율적으로 활용하기 어렵습니다.

1. **데이터 분포도가 낮은 경우**

성별처럼 두 가지 값만 있는 컬럼은 인덱스가 있어도 옵티마이저가 Full Table Scan을 선택할 수 있습니다.

인덱스를 타는 것보다 그냥 전체를 읽는 것이 더 빠를 수 있기 때문입니다.

---

### 인덱스의 단점

인덱스는 조회 성능을 높이지만, 단점 또한 존재합니다.

- INSERT / UPDATE / DELETE 시 인덱스도 함께 갱신을 해야 하므로 쓰기 성능이 저하됩니다.
- 인덱스 자체도 디스크 공간을 차지합니다.
- 인덱스가 너무 많으면 옵티마이저가 잘못된 인덱스를 선택할 수 있습니다.

이러한 단점들 때문에 인덱스는 조회가 빈번하고, 카디널리티가 높고, 쓰기보다 읽기가 많은 컬럼에 생성하는 것이 좋습니다.

### 쿼리 최적화

- **SELECT* 지양**

필요한 컬럼만 조회하면 네트워크 전송량을 줄이고, 커버링 인덱스를 활용할 가능성이 높아집니다.

- **페이징 쿼리 최적화**

```jsx
-- 데이터가 많아질수록 느려지는 방식 (OFFSET이 클수록 앞의 행을 모두 읽고 버림)
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 1000000;

-- No-Offset 방식 (마지막으로 읽은 ID를 기준으로 다음 데이터 조회)
SELECT * FROM users WHERE id > 1000000 ORDER BY id LIMIT 10;
```

- **fetch join**

fetch join을 통해 N+1 문제를 해결함으로 쿼리를 최적화할 수 있습니다.
