### 트랜잭션이란

트랜잭션은 데이터베이스에서의 논리적인 작업 단위입니다.

예를 들어 A가 B에게 계좌이체를 하는 경우에

- A 계좌에서 금액이 차감됨
- B 계좌에 금액이 추가됨

이 두 작업은 반드시 함께 성공하거나 함께 실패해야 하기 때문에 트랜잭션을 사용합니다.

---

### ACID

트랜잭션은 원자성, 일관성, 격리성, 지속성의 4가지 특성을 가집니다

- **A**tomicity(원자성) : 트랜잭션 내 작업은 모두 성공하거나 모두 실패해야 합니다.
- **C**onsistency(일관성) : 트랜잭션 전후로 데이터는 항상 일관된 상태를 유지해야 합니다.
- **I**solation(격리성) : 동시에 실행되는 트랜잭션들은 서로 간섭하지 않아야 합니다.
- **D**urability(지속성) : 트랜잭션이 한번 커밋되면 결과는 영구적으로 반영되어야 합니다.

---

### Spring 에서의 트랜잭션

Spring에서는 `@Transactional` 어노테이션을 통해 트랜잭션을 선언적으로 관리합니다.

메서드에 @Transactional 어노테이션을 붙이게 되면 해당 메서드는 트랜잭션으로 작동하고 메서드가 정상 종료되면 커밋, 런타임 예외가 발생하면 자동으로 롤백이 수행됩니다.

@Transactional 에서 자주 사용되는 옵션입니다.

- `readOnly = true` : 읽기 전용 트랜잭션 선언으로 불필요한 스냅샷 생성을 줄여 성능을 최적화합니다.
- `timeout` : 지정 시간 초과 시 롤백을 수행합니다.
- `rollbackFor` : 특정 예외에만 롤백을 적용합니다.

---

### 트랜잭션 전파 (Propagation)

트랜잭션이 이미 존재하는 상황에서 새로운 트랜잭션을 시작할 때 어떻게 처리할지를 정의하는 것이 전파 속성입니다.

- `REQUIRED` (default) : 기존 트랜잭션이 있으면 참여, 없으면 새로 생성합니다.
- `REQUIRES_NEW` : 항상 새로운 트랜잭션을 생성하고 기존 트랜잭션은 일시 중단합니다. 로그 저장과 같이 메인 트랜잭션의 성공이나 실패 여부와는 무관하게 독립적으로 커밋되어야 하는 경우에 사용합니다.
- `SUPPORTS` : 기존 트랜잭션이 있으면 참여하고 없으면 트랜잭션 없이 실행합니다.
- `NOT_SUPPORTED` : 트랜잭션 없이 실행하며 기존 트랜잭션은 일시 중단합니다.
- `NEVER` : 트랜잭션이 존재하면 예외를 발생시킵니다.

### 동시성 문제

여러 트랜잭션이 동시에 같은 데이터에 접근하면 다양한 문제가 발생할 수 있습니다.

1. Dirty Read
    
    다른 트랜잭션이 아직 커밋하지 않은 데이터를 읽는 경우입니다.
    
    트랜잭션 A가 데이터를 변경했지만 아직 커밋하지 않은 상태에서 트랜잭션 B가 그 값을 읽었을 때, A가 롤백된다면 B는 존재하지 않는 데이터를 읽은 것이 됩니다.
    
2. Non-Repeatable Read
    
    같은 트랜잭션 안에서 같은 데이터를 두 번 읽었을 때 값이 다른 경우입니다.
    
    첫 번째 조회와 두 번째 조회 사이에 다른 트랜잭션이 해당 데이터를 수정하고 커밋하게 될 경우 발생합니다.
    
3. Phantom Read
    
    같은 트랜잭션 안에서 같은 조건으로 조회했을 때 결과의 행 수가 달라지는 경우입니다.
    
    다른 트랜잭션이 새로운 행을 추가하거나 삭제할 경우 발생합니다.

### 트랜잭션 격리 수준 (Isolation Level)

이러한 동시성 문제들을 얼마나 방지할지 결정하는 것이 격리 수준입니다.

격리 수준이 높을수록 안전하지만 성능이 낮아지고, 낮을수록 성능은 좋아지지만 동시성 문제가 발생할 수 있게됩니다.

- `READ UNCOMMITED` - 가장 낮은 수준
- `READ COMMITTED`
- `REPEATABLE READ`
- `SERIALIZABLE`  - 가장 높은 수준

Spring에서는 `@Transactional(isolation = Isolation.READ_COMMITTED)` 의 형태로 격리 수준을 설정합니다.

### DB 락

동시성 문제를 해결하는 또 다른 방법으로 DB 락이 있습니다.

락은 특정 데이터에 대한 접근을 제한해 데이터의 정합성을 보장합니다.

- 공유 락 (Shared Lock / S-Lock) : 데이터를 읽을 때 사용하는데, 다른 트랜잭션이 읽을 수는 있지만 수정이 불가합니다.
- 배타 락 (Exclusive Lock / X-Lock) : 데이터를 쓸 때 사용하는데, 다른 트랜잭션이 읽기와 쓰기 모두 불가합니다.

---

### 낙관적 락과 비관적 락

실제 개발을 할때는 DB 락을 사용할 때의 전략이 중요합니다.

- 비관적 락 (Pessimistic Lock)

비관적 락은 충돌이 자주 발생할 것이라고 가정한 뒤, 데이터를 조회할 때부터 락을 거는 방식입니다.

예를 들어

```jsx
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT a FROM Account a WHERE a.id = :id")
Account findByIdWithLock(@Param("id") Long id);
```

이 경우에는 `SELECT ... FOR UPDATE` 쿼리가 발생하며 다른 트랜잭션은 해당 데이터에 접근하지 못하고 대기하게 됩니다.

데이터의 충돌이 잦은 경우 안전을 챙길 수 있지만, 락 대기로 인해서 성능이 저하될 수 있습니다.

- 낙관적 락 (Optimistic Lock)

낙관적 락은 충돌이 거의 없을 것이라고 가정한 뒤, 실제로 수정이 발생할 때 충돌 여부를 확인하는데, DB 락을 사용하지 않고 버전 컬럼(Version)을 주로 활용합니다.

예를 들어

```jsx
@Entity
public class Account {
    @Id
    private Long id;

    @Version
    private Long version;
}
```

이 경우에는 데이터를 수정할 때 조회 시점의 version과 현재의 version을 비교해, 다른 트랜잭션이 먼저 수정을 했다면 `OptimisticLockException` 이 발생하게 됩니다.

충돌이 잘 일어나지 않는 경우 성능이 좋지만, 충돌 시 재시도 로직을 직접 구현해야 합니다.

### 데드락 (Deadlock)

트랜잭션 A가 자원1을 점유하고 자원2를 기다리는 동시에, 트랜잭션 B가 자원2를 점유하고 자원1을 기다리는 교차 상황이 발생할 수 있습니다.

이처럼 두 트랜잭션이 서로의 락이 해제되기를 무한정 기다리게 되는 상황을 데드락이라고 합니다.

MySQL의 경우 데드락을 감지하면 한 트랜잭션을 자동으로 롤백해 교착 상태를 해소하게 됩니다.

이러한 데드락을 예방하기 위해서는

- 여러 테이블에 접근하는 경우 항상 동일한 순서로 접근
- 트랜잭션을 최대한 짧게 유지
- 락의 범위를 최소화

의 형태로 설계하는 것이 좋습니다.

---

### 커넥션 풀과의 관계

비관적 락이나 긴 트랜잭션은 Connection을 오래 점유하게 됩니다.

따라서 트랜잭션을 설계할 때는

- 트랜잭션 범위를 최소화 해 Connection 점유 시간을 줄인다.
- 외부 API 호출, 파일 처리 등은 트랜잭션 바깥에서 수행한다.
- 읽기 작업에는 `@Transactional(readOnly = true)` 를 적용해 불필요한 락을 피한다

와 같은 점들을 고려해 설계해야 합니다.
