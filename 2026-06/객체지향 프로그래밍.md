### 객체지향이란

객체지향 프로그래밍(OOP)은 프로그램을 객체(Object)들의 상호작용으로 설계하는 방식입니다.

절차지향 프로그래밍(AOP)이 데이터와 처리 로직을 분리해 순서대로 처리하는 방식이라면, 객체지향 프로그래밍은 데이터(속성)와 그 데이터를 처리하는 로직(메서드)을 하나의 객체로 묶어 관리합니다.

이를 통해 코드의 **재사용성**, **유지보수성**, **확장성**을 높일 수 있습니다.

---

### 객체지향의 특성

1. **캡슐화** (Encapsulation)

캡슐화는 객체의 속성과 메서드를 하나로 묶고 외부에는 필요한 부분만 노출하는 것입니다. 

private을 사용해 외부에서 직접 접근할 수 없도록 막고, 이를 변경하는 메서드는 public으로 열어 해당 메서드를 통해서만 private 값을 변경하도록 해 잘못된 값이 들어오는 것을 막고 상태를 관리할 수 있습니다.

1. **상속** (Inheritance)

상속은 기존 클래스의 속성과 메서드를 물려받아 새로운 클래스를 정의하는것입니다.

extends로 기존 클래스를 상속할 수 있습니다.

상속은 코드 재사용성을 높이지만 남용하게 되면 클래스 간 결합도가 높아져 변경에 취약해질 수 있습니다.

1. **다형성** (Polymorphism)

다형성은 인터페이스를 사용해 같은 타입이지만 실행 시점에 다른 동작을 하도록 합니다.

```jsx
public interface PaymentMethod {
    void pay(int amount);
}

public class CardPayment implements PaymentMethod {
    public void pay(int amount) {
        System.out.println("카드 결제: " + amount);
    }
}

public class CashPayment implements PaymentMethod {
    public void pay(int amount) {
        System.out.println("현금 결제: " + amount);
    }
}
```

예시 코드를 보면 CardPayment와 CashPayment 두 클래스 모두 PaymentMethod 인터페이스를 구현하고 있습니다.

이 때 실제로 어떤 구현체가 들어오는지에 따라 `pay()` 의 동작이 달라지게 됩니다.

1. **추상화** (Abstraction)

추상화는 객체의 공통적인 속성과 행위를 추출해 불필요한 세부사항은 감추고 필요한 부분만 정의하는 것입니다.

인터페이스나 추상 클래스를 통해 “What”만 정의하고, “How”는 구현체에 위임하게 됩니다.

---

### 추상클래스 vs 인터페이스

추상 클래스

- 다중 상속 불가 (단일 상속)
- 생성자를 가질 수 있음
- 일반 필드 가능
- 일반 메서드 + 추상 메서드
- “is-a” 관계, 공통 로직을 공유

인터페이스

- 다중 상속 가능 (다중 구현)
- 생성자를 가질 수 없음
- `public static final` 필드만 가능
- 추상 메서드 + default/static 메서드
- “can-do” 관계, 행위 규약 정의

추상 클래스는 관련된 클래스들이 공통 로직을 공유할 때, 인터페이스는 서로 다른 클래스들이 같은 행위를 보장해야 할 때 사용합니다.

---

### SOLID 원칙

객체 지향 설계를 하기 위한 5가지 원칙입니다.

1. **단일 책임 원칙** ( Single Responsibility Principle )

클래스는 하나의 책임만 가져야 합니다. → 즉 클래스를 변경하는 이유가 하나여야 합니다.

책임이 섞여 있으면 한쪽 로직을 수정할 때 다른 쪽에도 영향을 줄 위험이 커집니다.

1. **개방-폐쇄 원칙** ( Open-Closed Principle )

확장에는 열려있고, 변경에는 닫혀 있어야 합니다.

새로운 기능을 추가할 때 기존 코드를 수정하지 않고, 코드를 추가하는 것만으로 확장할 수 있어야 합니다.

인터페이스를 활용하면 새로운 기능이 추가되어도 기존 코드를 변경하지 않고 새로운 구현체만 추가할 수 있습니다.

1. **리스코프 치환 원칙** ( Liskov Substitution Principle )

자식 클래스는 부모 클래스를 대체할 수 있어야 합니다. → 즉 부모 타입으로 동작하는 곳에 자식 객체를 넣어도 프로그램이 정상 동작해야 합니다.

1. **인터페이스 분리 원칙** ( Interface Segregation Principle )

클라이언트는 자신이 사용하지 않는 인터페이스에 의존하지 않아야 합니다.

하나의 모든 기능이 포함된 거대한 인터페이스보다, 클라이언트가 필요한 기능별로 인터페이스를 분리하는 것이 좋습니다.

1. **의존 역전 원칙** ( Dependency Inversion Principle )

고수준 모듈은 저수준 모듈에 의존해서는 안되고 둘 다 추상화에 의존해야 합니다.

Spring에서는 의존성 주입(DI)을 통해 의존 역전 원칙을 구현 수준에서 실현하고 있습니다.

---

### Spring과 SOLID

Spring 내부 동작은 SOLID 원칙에 따라 작동합니다.

- DI 컨테이너는 의존 역전 원칙(DIP)를 실현하기 위한 도구입니다. Service가 구체 클래스가 아닌 Repository(인터페이스)에 의존하게 되고, 실제 구현체는 Spring이 런타임에 주입합니다.
- 다형성은 개방 폐쇄 원칙(OCP)에 기반합니다. 새로운 구현체를 추가해도 이를 사용하는 코드는 변경되지 않습니다.
- AOP는 단일 책임 원칙(SRP)를 지원합니다. 트랜잭션, 로깅 같은 부가 기능을 비즈니스 로직과 분리함으로써 핵심 로직 클래스가 하나의 책임만 가지도록 도와줍니다.

---

### 결합도와 응집도

- 결합도 (Coupling)

결합도는 모듈 간 의존하는 정도를 나타냅니다.

결합도가 낮을수록 한 모듈의 변경이 다른 모듈에 영향을 적게 줍니다.

- 응집도 (Cohesion)

응집도는 하나의 모듈 내부 요소들이 얼마나 관련 있는 책임으로 묶여 있는지를 나타냅니다. 응집도가 높을수록 하나의 책임에 집중된 것입니다.

좋은 설계는 **낮은 결합도, 높은 응집도** 를 지향합니다.

SRP는 응집도를 높이는것에, DIP는 결합도를 낮추는 것에 기여합니다.
