### DB와 통신하는 과정

어플리케이션이 데이터베이스에 데이터를 조회하거나 저장하려면 먼저 DB와 연결(Connection)을 맺어야 합니다.

사용자가 로그인 요청을 보낸 경우

- 로그인 요청
- DB 연결 생성
- 사용자 조회
- 결과 반환
- DB 연결 종료

의 과정이 수행됩니다.

---

### DB 연결 생성 작업

DB Connection은 다음과 같은 과정을 거칩니다

애플리케이션 → TCP 연결 생성 → DB 인증 수행 → 세션 생성 → Connection 생성 완료

Connection을 생성할 때마다

- TCP Handshake
- DB 인증
- 세션 생성

이 수행되기 때문에 매 요청마다 생성을 반복하게 되면 성능이 크게 떨어집니다.

---

### Connection Pool

이를 해결하기 위해서 미리 여러 개의 DB Connection을 생성해두고 관리하는 공간인 Connection Pool을 사용합니다.

요청이 들어오면 새로운 Connection을 만드는 것이 아니라, Pool에 있는 Connection을 빌려 사용하게 됩니다.

서버시작 → Connection n개 생성

요청 → Pool에서 Connection 획득 → SQL 실행 → Pool에 반납

의 과정을 거치게 되어 Connection 생성 비용이 발생하지 않게 됩니다.

따라서 미리 생성된 Connection을 많은 요청들이 재사용 하게 되며

- 응답속도 향상
- DB 부하 감소
- 서버 자원 절약

의 효과를 얻을 수 있습니다.

---

### HikariCP

Spring Boot는 기본적으로 HikariCP을 사용합니다.

HikariCP는 현재 가장 많이 사용되는 JDBC Connection Pool 구현체 중 하나로,

- 빠른 성능
- 낮은 메모리 사용량
- 안정성

의 장점을 가지고 있습니다.

### HikariCP 주요 설정

```jsx
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 10
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
```

예시로 자주 사용되는 설정들입니다.

`maximum-pool-size`  : Pool이 가질 수 있는 최대 Connection의 개수를 뜻합니다.

`minimum-idle` : 항상 유지할 최소 Connection의 수로, Connection이 줄어들어도 최소 몇개를 유지할지 설정합니다.

`connection-timeout` : Connection을 빌리기 위해 기다리는 시간으로, 설정된 시간 안에 빈 Connection이 없다면 예외가 발생합니다.

`idle-timeout` : 사용되지 않는 Connection 제거 시간으로, 설정된 시간동안 사용되지 않았다면 제거합니다.

`max-lifetime` : Connection의 최대 수명으로 설정된 시간이 지나면 재생성을 합니다. DB서버가 오래된 Connection을 강제로 끊는 경우를 대비하기 위함입니다.

---

### Connection Pool을 크게 하면 무조건 좋을까?

Pool Size를 크게 잡으면 성능이 좋아질 것 같지만 그렇지 않습니다.

Connection 하나당

- 메모리 사용
- DB 세션 사용
- CPU 사용

이 일어나기 때문에 오히려 Connection이 너무 많다면

- Connection끼리 경쟁
- Context Switching 증가
- DB 과부하

의 문제가 발생할 수 있습니다.

---

### Pool Size 설정

그래서 보통은

- CPU 코어 수
- DB 성능
- 트래픽
- 쿼리 수행 시간

을 고려해 모니터링하며 Pool Size를 조정하게 됩니다.

커넥션 풀이 부족하게 된다면 요청 대기가 늘어나게 되고, Connection timeout 예외가 발생하게 됩니다.

`Connection is not available, request timed out`  의 형태로 로그가 남습니다.

---

### 문제 상황

1. Connection Leak

Connection을 반납하지 않는 경우가 있습니다.

커넥션을 생성하고, `connection.close();` 를 호출하지 않는다면 Pool 은 해당 Connection이 계속 사용중이라고 판단해 Pool 고갈이 발생할 수 있습니다.

이는 Spring JPA나 JdbcTemplate을 사용하게 되면 자동 관리를 할 수 있습니다.

1. 긴 트랜잭션

하나의 트랜잭션이 외부API호출, 파일 업로드, 대용량 작업 등을 수행하게 된다면 Connection을 오래 점유하게 됩니다.

그렇다면 결과적으로 Pool 부족 현상이 발생할 수 있습니다.

1. N+1 문제

JPA를 사용하면서 발생하는 대표적인 문제입니다.

JPA의 N+1 문제는 연관관계가 있는 엔티티를 조회할 때 조회된 개수 N개 만큼의 쿼리가 추가로 발생하게 되는 경우 발생합니다.

따라서 N+1 문제가 발생하게 되면 Connection 사용 시간이 길어지기 때문에 쿼리 최적화를 통해 해결해야 합니다.
