### 로그인 기능 구현

로그인 기능을 구현할 때는 보통 JWT 방식을 많이 사용합니다.

JWT 방식으로 로그인에 성공하면

- Access Token
- Refresh Token

이 발급됩니다.

Access Token의 경우 로그인을 해야 사용할 수 있는 API들에서 클라이언트가 API요청 시 `Authrization` 헤더에 `Bearer <token>` 형식으로 담아 백엔드에 보내, 유효한 사용자인지를 확인하기 위해 사용합니다.

그래서 Access Token의 경우 탈취 위험이 있기 때문에 1시간 등 짧은 시간동안 활용하고, 

Refresh Token의 경우 Access Token이 만료되어도 로그인을 유지할 수 있도록 새로운 Access Token을 발급받기 위해 사용합니다.

---

### 그렇다면 Refresh Token은 어떻게 저장할까

Refresh Token을 DB에 저장할 수도 있습니다.

하지만 로그인과 토큰 재발급 기능은 매우 빈번하게 발생합니다.

Access Token을 재발급 받을 때마다 DB에서

- Refresh Token 조회
- 검증
- 갱신

의 과정을 수행해야 하고, 사용자가 많아질수록 반복적인 요청이 DB에 부하가 될 수 있습니다.

---

### Redis란

Redis는 인메모리 데이터 스토어입니다.

관계형 데이터베이스(RDBMS)와는 달리 Redis는 `Key - Value`의 형태로 데이터를 저장합니다.

또한 Redis는 데이터를 디스크가 아닌 메모리(RAM)에 저장하기 때문에 읽기와 쓰기 속도가 매우 빠릅니다.

따라서 사용자 정보나 게시글처럼 보존되어야 하는 정보는 RDB에, 캐시나 빠른 조회가 필요한 데이터는 Redis에 올려 사용합니다.

---

### 로그인에 Redis를 사용하는 이유

Refresh Token의 경우

- 자주 조회된다
- 자주 갱신된다
- 사용자 데이터와 복잡한 관계를 가지지 않는다
- 만료시간(TTL)이 존재한다

는 특성이 있습니다.

따라서 로그인 기능 중 Refresh Token을 활용해 Access Token을 재발급 할 때는 Redis를 활용하는 것이 좋습니다.

---

### 전체 로그인 기능 흐름

이를 바탕으로 전체 로그인 기능의 흐름을 간략하게 살펴보면

- 최초 로그인 → AccessToken/RefreshToken 발급
- Redis에는 `userId: refreshToken` 형식으로 저장
- API 요청 시 AccessToken 검증
- AccessToken 만료시 → RefreshToken으로 재발급 요청
- Redis에서 RefreshToken 조회 및 검증
- 검증이 이루어졌다면 새로운 AccessToken 발급
- 로그아웃 시 Redis에서 RefreshToken 삭제

---

### 실제 개발 시 고려해야할 사항

실제 로그인 기능을 구현할 때 고려해야할 부분이 있습니다.

1. Redis에 Refresh Token을 저장하는 방법

Redis에 Refresh Token을 저장할 때는 Key를 어떻게 설계하는지가 중요합니다.

Key를 단순히 사용자ID로 설정하면, 다른 용도의 데이터와 충돌할 수 있습니다.

따라서 prefix를 붙여 설계하는 방식을 사용합니다.

- Key : `RT : {userId}`
- Value : Refresh Token문자열

또한 Redis는 데이터에 TTL (Time To Live)을 설정할 수 있습니다.

이 때 TTL을 Refresh Token의 만료 시간과 동일하게 설정하면, 만료된 Refresh Token이 Redis에서 자동으로 삭제되어 별도의 삭제 로직 없이 저장 공간을 효율적으로 관리할 수 있습니다.

1. RTR ( Refresh Token Rotation )

Access Token을 재발급 할 때 Refresh Token도 함께 교체하는 방식입니다.

Refresh Token을 한 번 사용하면 즉시 폐기하고, 새로운 Refresh Token을 발급해 Redis의 key에 덮어씁니다.

이미 사용된 Refresh Token으로 재발급 요청이 들어온다면, Redis에 저장된 값과 일치하지 않기 때문에 Refresh Token이 탈취되었다고 판단해 해당 사용자의 토큰을 무효화할 수 있게 됩니다.

1. 로그아웃 처리

로그아웃 시에는 단순히 클라이언트에서 토큰을 삭제하는 것만으로는 부족합니다.

클라이언트가 토큰을 삭제하더라도, 해당 Access Token의 만료 시간이 남아있다면 탈취된 토큰으로 API를 호출하는 것이 가능하기 때문입니다.

따라서 로그아웃 시에는

- Redis에서 Refresh Token 삭제
- 아직 만료되지 않은 Access Token을 블랙리스트로 Redis에 등록

블랙리스트에 등록된 Access Token은 이후 API 요청 시 필터에서 차단되며

TTL을 Access Token의 잔여 만료 시간으로 설정하면 만료와 동시에 자동으로 삭제됩니다.

### 보안 고려사항

보안 고려사항에 대해서는 우선 간단하게 살펴본 뒤 추후 좀더 깊게 공부해 보려고 합니다.

1. HTTPS 적용

JWT는 Base64로 인코딩 된 값이기 때문에 HTTP환경에서는 네트워크 상에서 토큰이 노출될 수 있습니다.

따라서 HTTPS를 적용해 전송 구간을 암호화 해야 합니다.

1. Refresh Token 저장 위치

클라이언트에서 Refresh Token을 어디에 저장하는지도 중요합니다.

- localStorage : JavaScript로 접근이 가능하기 때문에 XSS 공격에 취약합니다.
- HttpOnly Cookie : JavaScript로 접근할 수 없어 XSS에는 안전하지만, CSRF 공격에 대한 별도 대응이 필요합니다.

일반적으로는 HttpOnly Cookie에 저장하는 방식이 권장됩니다.

1. Access Token 유효 시간

Access Token의 유효 시간이 길수록 탈취 시 피해가 커지기 때문에

통상적으로 15분에서 1시간 사이로 설정합니다.
