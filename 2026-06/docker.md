### Docker

개발을 하다보면 개발자의 로컬환경에서는 완벽하게 동작하던 애플리케이션이 서버에 올라가면 동작하지 않는 상황에 마주할 수 있습니다.

이러한 이유는 환경 차이에서 오는데, Java 버전이 다르거나, 운영체제가 다르거나, 설치된 라이브러리 버전이 달라 발생할 수 있습니다.

이를 해결하기 위해 Docker를 사용합니다.

애플리케이션과 그것이 실행되는데 필요한 모든 환경을 하나의 패키지로 묶어, 어떤 환경이라도 동일하게 동작하도록 합니다.

---

### 컨테이너

Docker는 컨테이너라는 규격화된 단위로 포장해 어떤 서버에서든 동일하게 실행하도록 합니다.

컨테이너 이전에도 환경 격리 문제를 해결하기 위해 가상머신(VM)을 사용했습니다.

VM은 하나의 컴퓨터 위에 또 다른 가상의 컴퓨터를 만드는 방식입니다. 가상의 컴퓨터는 독립된 운영체제를 가지는데, 따라서 무겁고 시작하는데 시간이 오래 걸립니다.

컨테이너는 다른 접근 방식을 취합니다.

OS를 새로 올리는 대신 호스트 OS의 커널을 공유하면서 프로세스 수준에서만 격리합니다.

이것은 리눅스 커널의 두 가지 기능 덕분에 가능한데,

namespace를 통해 프로세스가 볼 수 있는 자원의 범위를 격리합니다. 컨테이너 안의 프로세스는 자신만의 독립된 파일 시스템, 네트워크, 프로세스 목록을 가진 것처럼 동작합니다.

cgroup은 프로세스가 사용할 수 있는 CPU, 메모리 같은 자원의 양을 제한합니다.

이 두 기능을 활용해 OS없이도 격리된 실행 환경을 만들어낸 것이 컨테이너입니다.

### 이미지와 컨테이너

- **이미지**

이미지는 컨테이너를 만들기 위한 설계도입니다. 애플리케이션 코드, 런타임, 라이브러리, 설정 등이 모두 담긴 읽기 전용 패키지입니다.

한 번 만들어진 이미지는 변경되지 않습니다.

- **컨테이너**

컨테이너는 이미지를 실제로 실행한 인스턴스입니다. 이미지를 기반으로 실제 프로세스가 동작하는 공간입니다.

이미지는 클래스, 컨테이너는 인스턴스의 개념이라 하나의 이미지로 동일한 컨테이너를 여러 개 실행할 수 있습니다.

---

### 이미지의 레이어 구조

이미지는 단일 파일이 아니라 여러 레이어가 순서대로 쌓인 구조입니다.

예를들어 Spring Boot의 애플리케이션 이미지를 만든다면

- Java 런타임 레이어
- 의존성 라이브러리 레이어
- 애플리케이션 코드 레이어

와 같은 형식으로 레이어가 쌓입니다.

이러한 레이어 구조가 중요한 이유는 캐싱에 있습니다.

이미지를 다시 빌드할 때 변경된 레이어부터만 새로 빌드하고, 그 아래의 레이어는 캐시 재사용을 합니다. 코드만 바꿨다면 Java 런타임과 라이브러리는 다시 받지 않기 때문에 빌드 속도가 크게 빨라지게 됩니다.

또한 여러 이미지가 동일한 레이어를 공유할 수 있습니다. Java 런타임 레이어를 이미 가지고 있다면, 다른 이미지를 받을 때 해당 레이어는 다시 받지 않게되고, 디스크 공간도 절약할 수 있습니다.

### 서비스 배포와 Docker

Docker의 등장 이전 배포 방식은 서버에 직접 접속해 환경을 설치하고 애플리케이션을 실행하는 방식이었습니다.

서버가 여러대라면 같은 과정을 반복해야 했습니다.

Docker를 사용하면 이미지 하나만 있으면 어떤 서버에서든 완전히 동일한 환경에서 애플리케이션을 실행할 수 있게됩니다.

또한 이미지는 버전 관리가 가능하기 때문에 새 버전을 배포했다가 문제가 생기면 이전 버전의 이미지로 즉시 롤백할 수 있습니다.

---

### Docker Compose

실제 서비스는 하나의 컨테이너만으로 동작하지 않습니다.

Spring Boot, MySql, Redis 등등이 함께 동작해야 한다면, 모든 컨테이너가 서로 통신하며 실행되어야 합니다.

Docker Compose는 이런 멀티 컨테이너 환경을 하나의 파일로 정의하고 한 번에 관리할 수 있게 해주는 도구입니다.

명령어 하나로 정의된 모든 컨테이너를 동시에 시작하거나 중지할 수 있습니다.

---

### 볼륨

컨테이너는 기본적으로 상태를 저장하지 않습니다.

컨테이너를 삭제하면 그 안에 저장된 데이터도 함께 사라집니다. 따라서 DB 컨테이너를 재시작하면 데이터가 모두 날아가는 상황이 생길 수 있습니다.

이러한 문제를 해결하는 것이 볼륨인데, 볼륨은 컨테이너 외부 즉 호스트 파일 시스템에 데이터를 저장하는 공간입니다.

컨테이너가 삭제되어도 볼륨에 저장된 데이터느 유지됩니다. 새 컨테이너를 띄우면서 같은 볼륨을 연결하면 데이터는 그대로 이어집니다.

### Dockerfile 작성

아래는 Spring Boot의 Dockerfile 예시입니다.

```jsx
# 베이스 이미지 지정 (Java 17 포함된 경량 이미지)
FROM openjdk:17-jdk-slim

# 작업 디렉토리 설정
WORKDIR /app

# 빌드된 jar 파일을 컨테이너 안으로 복사
COPY build/libs/myapp-0.0.1-SNAPSHOT.jar app.jar

# 컨테이너가 사용할 포트 명시 (문서 목적, 실제 포트 개방은 run 명령어에서)
EXPOSE 8080

# 컨테이너 시작 시 실행할 명령어
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### Docker 명령어

아래는 자주 쓰이는 Docker 명령어들입니다.

```jsx
# 이미지 빌드 (현재 디렉토리의 Dockerfile 사용)
docker build -t myapp:latest .

# 컨테이너 실행
# -d : 백그라운드 실행
# -p 8080:8080 : 호스트 포트:컨테이너 포트 매핑
docker run -d -p 8080:8080 myapp:latest

# 실행 중인 컨테이너 목록 확인
docker ps

# 컨테이너 로그 확인
docker logs [컨테이너ID]

# 컨테이너 중지 및 삭제
docker stop [컨테이너ID]
docker rm [컨테이너ID]

# 이미지 목록 확인
docker images

# 이미지 삭제
docker rmi myapp:latest
```

### Docker-Compose

Docker Compose는 .yml파일로 관리하는데 아래는 그 예시입니다.

```jsx
# docker-compose.yml
version: '3.8'

services:
  app:
    build: .
    ports:
      - "8080:8080"
    environment:
      - SPRING_DATASOURCE_URL=jdbc:mysql://db:3306/mydb
      - SPRING_REDIS_HOST=redis
    depends_on:
      - db
      - redis

  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: password
      MYSQL_DATABASE: mydb
    volumes:
      - mysql_data:/var/lib/mysql

  redis:
    image: redis:7.0

volumes:
  mysql_data:
```

이 때 docker compose는

`docker compose up -d` 

`docker compose down` 

명령어를 사용해 관리합니다.
