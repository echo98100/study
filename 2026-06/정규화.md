### 정규화

정규화는 데이터 중복을 제거하고 데이터를 논리적으로 올바른 구조로 분리하는 과정입니다.

정규화의 핵심 원칙은 하나의 테이블은 하나의 주제만 담아야 한다는 것입니다.

정규화는 단계별로 진행되며 각 단계를 정규형이라고 합니다.

---

### 1차 정규형 (1NF)

**원칙 : 하나의 컬럼에는 하나의 값만 저장한다**

1NF를 위반하는 경우는 하나의 셀에 여러 값이 들어가는 경우입니다.

```jsx
위반 예시
┌────────┬──────┬──────────────────┐
│ 주문ID  │  고객 │     상품목록       │
├────────┼──────┼──────────────────┤
│   1    │  관영 │   노트북, 마우스    │  ← 하나의 셀에 여러 값
└────────┴──────┴──────────────────┘

1NF 적용 후
┌────────┬──────┬────────┐
│ 주문ID  │  고객 │.  상품  │
├────────┼──────┼────────┤
│   1    │ 관영  │  노트북  │
│   1    │ 관영  │  마우스  │
└────────┴──────┴────────┘
```

---

### 2차 정규형 (2NF)

**원칙 : 1NF를 만족하면서 모든 컬럼이 기본키 전체에 종속되어야 한다**

기본키가 복합키일때 적용됩니다.

기본키의 일부에만 종속되는 컬럼을 분리하는데, 이를 부분 함수 종속 제거라고 합니다.

```jsx
위반 예시 (기본키: 주문ID + 상품ID)
┌────────┬───────┬──────┬──────────┐
│ 주문ID │ 상품ID │ 수량 │ 상품명   │
├────────┼───────┼──────┼──────────┤
│   1    │  101  │  2   │ 노트북   │  ← 상품명은 상품ID에만 종속
│   1    │  102  │  1   │ 마우스   │
└────────┴───────┴──────┴──────────┘

2NF 적용 후 → 테이블 분리
주문상품 테이블              상품 테이블
┌────────┬───────┬──────┐  ┌───────┬──────────┐
│ 주문ID │ 상품ID │ 수량 │  │ 상품ID │ 상품명   │
└────────┴───────┴──────┘  └───────┴──────────┘
```

---

### 3차 정규형 (3NF)

**원칙 : 2NF를 만족하면서 기본키가 아닌 컬럼이 다른 일반 컬럼에 종속되지 않아야 한다.**

이를 이행적 함수 종속이라고 합니다.

```jsx
위반 예시 (기본키: 주문ID)
┌────────┬────────┬────────────┬──────────┐
│ 주문ID │ 고객ID │ 고객이메일  │ 배송지   │
├────────┼────────┼────────────┼──────────┤
│   1    │   5    │ gy@naver   │ 부산     │  ← 고객이메일은 고객ID에 종속
│   2    │   5    │ gy@naver   │ 서울     │     (주문ID → 고객ID → 고객이메일)
└────────┴────────┴────────────┴──────────┘

3NF 적용 후 → 테이블 분리
주문 테이블                  고객 테이블
┌────────┬────────┬──────────┐  ┌────────┬────────────┐
│ 주문ID │ 고객ID │ 배송지   │  │ 고객ID │ 고객이메일  │
└────────┴────────┴──────────┘  └────────┴────────────┘
```

---

### 정규화의 장단점

**장점**

- 데이터 중복이 줄어들어 저장 공간이 절약됩니다.
- 수정이 필요할 경우 한 곳만 바꾸면 되어 데이터 일관성이 보장됩니다.
- 이상 현상이 제거됩니다.

**단점**

- 테이블이 여러 개로 분리되어 조회 시 JOIN이 많아집니다.
- JOIN이 많아질수록 쿼리가 복잡해지고 성능이 저하될 수 있습니다.

---

### 반정규화

반정규화는 정규화된 테이블을 의도적으로 합치거나 중복을 허용해 조회 성능을 높이는 과정입니다.

정규화를 거스르는 것처럼 보이지만 이상 현상보다 조회 성능이 더 중요한 상황에서 의도적으로 선택하는 트레이드오프입니다.

정규화가 쓰기 최적화 (데이터 정합성 보장) 이라면, 반정규화는 읽기 최적화 (조회 성능 향상) 입니다.

---

### 반정규화가 필요한 상황

반정규화는 다음과 같은 경우에 고려할 수 있습니다.

- 조회가 압도적으로 많고 쓰기는 적은 경우
- 특정 쿼리에서 JOIN이 너무 많아 성능 문제가 발생하는 경우
- 실시간으로 집계가 필요한 통계 데이터의 경우

---

### 반정규화 기법

- **컬럼 중복 허용**

JOIN을 줄이기 위해 다른 테이블의 컬럼을 가져와 중복 저장합니다.

```jsx
정규화된 구조 (조회 시 JOIN 필요)
주문 테이블: 주문ID, 고객ID
고객 테이블: 고객ID, 고객명

반정규화 적용 (JOIN 없이 조회 가능)
주문 테이블: 주문ID, 고객ID, 고객명  ← 고객명 중복 저장
```

위의 예시 상황에서 고객명이 바뀌면 주문 테이블도 함께 수정해야 하는 부담이 생기지만, 주문 내역 조회 시 매번 고객 테이블과 JOIN하지 않아도 됩니다.

- **테이블 병합**

자주 함께 조회되는 두 테이블을 하나로 합칩니다.

1:1 관계이거나, 두 테이블을 분리한 상태로는 항상 JOIN해서 사용하는 경우에 적용합니다.

- **집계 테이블 별도 관리**

통계나 집계 데이터를 매번 계산하지 않고 별도 테이블에 미리 계산해서 저장합니다.

```jsx
정규화된 방식
→ 주문 건수를 조회할 때마다 전체 주문 테이블을 COUNT

반정규화 적용
→ 사용자별 주문 건수를 별도 테이블에 미리 저장
→ 주문이 생길 때마다 해당 값을 갱신
→ 조회 시 집계 없이 바로 반환
```
