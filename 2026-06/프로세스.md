### 프로세스란

프로세스는 실행 중인 프로그램입니다.

프로그램은 디스크에 저장된 코드 파일이지만, 실행되는 순간 운영체제가 메모리를 할당하고 자원을 부여하는데, 이 실행 단위를 프로세스라고 합니다.

각 프로세스는 독립된 메모리 공간을 가지기 때문에 한 프로세스가 죽어도 다른 프로세스에 영향을 주지 않습니다.

---

### 스레드란

스레드는 프로세스 안에서 실행되는 작업 단위입니다.

하나의 프로세스는 여러개의 스레드를 가질 수 있으며, 같은 프로세스 안의 스레드들은 메모리를 공유합니다.

스레드는 각자 독립적인 Stack을 가지지만, Code / Data / Heap은 프로세스 내의 모든 스레드가 공유합니다.

이 공유 구조 덕분에 스레드 간 데이터 전달이 빠르지만 동시에 같은 데이터를 수정하면 동시성 문제가 발생할 수 있습니다.

동시성 문제를 해결하기 위해서 `synchronized`로 임계 구역을 설정하거나, Thread-safe 자료구조를 사용하기도 합니다.

또는 DB레벨에서 낙관적 락/ 비관적 락을 활용하기도 합니다.

웹 서버는 일반적으로 멀티스레드 방식을 사용합니다.

### Context Switching

CPU는 한 번에 하나의 작업만 처리할 수 있습니다.

여러 프로세스나 스레드가 동시에 실행되는 것 처럼 보이는 이유는 CPU가 매우 빠르게 작업을 번갈아 처리하기 때문입니다.

이때 현재 작업의 상태를 저장하고 다음 작업의 상태를 불러오는 과정을 Context Switching이라고 합니다.

Context Switching 자체도 CPU의 자원을 소모하는데, 스레드가 너무 많으면 실제 작업보다 Context Switching에 더 많은 자원이 쓰이는 상황이 발생합니다.

### Java의 스레드

Java는 기본으로 멀티스레드를 지원합니다.

Spring Boot의 내장 Tomcat은 기본적으로 요청마다 스레드를 하나씩 할당해 처리하고 기본 스레드 풀의 크기는 200개입니다.

이때 요청이 스레드 수를 초과하면 요청은 대기 큐에서 기다리게 됩니다.

Java에서는 멀티스레드 환경의 공유 자원 문제를 해결하기 위해 synchronized, volatile, java.util.concurrent 패키지를 사용합니다.

### 동기 vs 비동기

- **동기 (Synchronous)**

동기는 작업을 요청한 뒤, 그 결과가 올 때까지 기다리는 방식입니다.

요청자 → 작업 요청 → [기다림] → 결과 수신 → 다음 작업

순으로 진행되는데, 결과를 받은 후에 다음 작업으로 넘어가기 때문에 순서가 보장되고 흐름이 직관적입니다.

- **비동기 (Asynchronous)**

비동기는 작업을 요청한 뒤, 결과를 기다리지 않고 다음 작업을 계속 진행하는 방식입니다.

요청자 → 작업 요청 → 다음 작업 진행 → … → 추후 결과 수신

결과가 오면 콜백이나 이벤트로 처리하게 되는데 대기 시간 없이 다른 작업을 처리할 수 있어 효율적입니다.

---

### 블로킹 vs 논블로킹

블로킹/논블로킹은 동기/비동기와 관점이 다릅니다.

동기/비동기가 결과를 누가, 언제 처리하는가의 작업 흐름 관점이라면 블로킹/논블로킹은 제어권이 호출한 쪽에 있는가, 호출된 쪽에 있는가 하는 제어권의 관점입니다.

- **블로킹 (Blocking)**

호출한 쪽이 호출된 작업이 끝날 때까지 제어권을 넘겨주고 대기합니다.

Thread A → 작업 호출 → 제어권 넘김 → [멈춤] → 작업 완료 → 제어권 반환 → 재개

이렇게 작업을 호출하고 제어권이 넘어가면 스레드는 아무것도 못하고 멈춰있는 상태가 됩니다.

- **논블로킹 (Non-Blocking)**

호출한 쪽이 제어권을 유지한 채로 다른 작업을 계속할 수 있습니다.

Thread A → 작업 호출 → 제어권 유지 → 다른 작업 지속 → 추후 결과 확인

### 조합

동기/비동기와 블로킹/논블로킹을 활용한 네가지 조합이 가능합니다.

| 조합 | 설명 | 예시 |
| --- | --- | --- |
| 동기 + 블로킹 | 요청 후 결과 올 때까지 스레드가 멈춤 | 일반적인 JDBC 쿼리 |
| 동기 + 논블로킹 | 결과를 직접 처리하지만 스레드는 안 멈춤. 계속 완료 여부를 확인 | 게임 루프에서 폴링 방식 |
| 비동기 + 블로킹 | 비동기로 요청했지만 결과를 받을 때까지 스레드가 멈춤 | `Future.get()` 호출 |
| 비동기 + 논블로킹 | 요청 후 스레드가 멈추지 않고, 결과는 콜백으로 처리 | Spring WebFlux, Node.js |

전통적인 Spring MVC에서 쓰이는 동기+블로킹 조합이나, Spring WebFlux에서 쓰이는 비동기+논블로킹 조합이 대표적입니다.

- **Spring MVC (동기 + 블로킹)**

요청마다 스레드를 하나 할당해 처리합니다. 스레드가 DB 응답을 기다리는 동안 해당 스레드는 블로킹 상태로 대기합니다.

요청이 많아지면 스레드가 모두 대기 상태에 빠져 더 이상 새 요청을 받지 못하는 상황이 생길 수 있습니다.

- **Spring WebFlux (비동기 + 논블로킹)**

소수의 스레드로 많은 요청을 처리합니다. DB 응답을 기다리는 동안 스레드를 다른 요청 처리에 활용합니다.

I/O 대기가 많은 서비스(외부 API 다수 호출, 실시간 스트리밍 등)에서 효율적입니다.

다만 코드 복잡도가 높고, JPA 같은 블로킹 기반 라이브러리와 함께 쓰기 어렵다는 단점이 있습니다.

### @Async

Spring MVC 환경에서도 `@Async` 어노테이션을 통해 특정 작업만 비동기로 처리할 수 있습니다.

```jsx
@Service
public class EmailService {

    @Async
    public void sendEmail(String to) {
        // 별도 스레드에서 실행됨
        // 호출한 쪽은 이 메서드가 끝날 때까지 기다리지 않음
    }
}
```

`@Async`를 사용하면 해당 메서드는 별도의 스레드 풀에서 실행되고, 호출한 쪽은 바로 다음 작업으로 넘어갑니다.

주의할 점은 AOP 내부 호출 문제와 동일하게 같은 클래스 내부에서 호출하면 프록시를 거치지 않기 때문에 `@Async`가 적용되지 않습니다.
