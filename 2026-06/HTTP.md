### 브라우저의 URL

브라우저에서 URL을 입력하게 된다면

URL 입력 → DNS 조회 → TCP 연결 (3-way handshake) → HTTP 요청 → 서버 처리 → HTTP 응답 → 브라우저 렌더링

의 순서로 진행됩니다. 각 단계별로 살펴보자면

### DNS (Domain Name System)

브라우저는 도메인 이름을 IP 주소로 변환해야 합니다.

이를 담당하는 것이 DNS입니다.

DNS는

- 브라우저 캐시 확인
- OS 캐시 확인
- 로컬 DNS 서버 조회
- 루트 DNS
- TLD DNS(`.com` )
- 권한 DNS 서버
- IP 반환

의 순서로 동작하는데, 한번 조회한 결과는 TTL 동안 캐시에 저장되어 동일한 도메인에 대한 반복 조회를 줄입니다.

### TCP와 UDP

DNS를 통해 IP 주소를 얻었다면 해당 서버와 TCP 연결을 맺어야 합니다.

먼저 TCP와 UDP에 대해서 알아보겠습니다.

- TCP (Transmission Control Protocol)

TCP는 IP 규칙으로만 통신하기에는 불안정 했던 단점 ( 패킷 순서가 꼬이거나, 패킷이 유실되는 경우 )들을 커버해 패킷 전송을 제어하여 신뢰성을 보장하는 프로토콜입니다.

TCP는 규칙대로 올바르게 패킷이 도착했는지 정확히 누구에게 전달되어야 하는지를 따져 신뢰성 있는 통신이 필요할 때 사용됩니다.

이를 위해 3-Way Handshake 를 사용하는데 이는 조금 뒤에 자세히 보겠습니다.

- UDP (User Datagram Protocol)

UDP는 데이터그램 방식을 사용하는 프로토콜로 각각의 패킷 간의 순서가 존재하지 않는 독립적인 패킷을 사용합니다.

UDP는 신뢰성 확보를 위해 거치던 TCP의 과정을 거치지 않기 때문에 속도가 더 빠릅니다.

그래서 실시간 영상 스트리밍처럼 고용량 데이터를 다루는 곳에 주로 이용됩니다.

최근에는 HTTP/3가 UDP를 채택했는데, UDP는 개발자의 구현에 따라 TCP와 비슷한 수준의 신뢰성 기능을 가질 수도 있습니다.

---

### 3-Way Handshake

TCP가 연결을 맺을 때 거치는 3-way handshake 과정을 알아보겠습니다.

간단하게 보면

- 클라이언트 → SYN → 서버
- 클라이언트 ← SYN - ACK ← 서버
- 클라이언트 → ACK → 서버

의 과정을 통해 양쪽이 서로 데이터를 주고받을 준비가 되었음을 확인한 뒤에 연결이 수립되게 됩니다.

자세하게 살펴보자면 먼저

- 클라이언트는 서버에 접속을 요청하는 SYN 패킷을 보냅니다. 이때 클라이언트는 SYN을 보내고 SYN/ACK 응답을 기다리는 SYN_SENT 상태가 됩니다.
- 서버는 SYN요청을 받고 클라이언트에게 요청을 수락한다는 ACK와 SYN flag가 설정된 패킷을 발송하고 서버가 다시 ACK로 응답하기를 기다립니다. 이때 서버는 SYN_RECEIVED 상태가 됩니다
- 마지막으로 클라이언트가 서버에게 ACK를 보내게 되면 이후로부터는 연결이 이루어지고 데이터가 오가게 됩니다.

연결을 종료할 때는 4-way handshake를 통해 양쪽이 순서대로 연결을 끊게됩니다.

### HTTP

HTTP (HyperText Transfer Protocol)는 클라이언트와 서버가 데이터를 주고받기 위한 프로토콜입니다.

요청(Request)과 응답(Response)으로 이루어지며, 기본적으로 **stateless(무상태)** 특성을 가집니다.

이때 무상태란

서버는 이전 요청을 기억하지 않고, 매 요청을 독립적으로 처리하는 것입니다.

이 때문에 로그인 상태를 유지하기 위해서 JWT나 세션/쿠키가 필요하게 됩니다.

---

### HTTP 메서드

HTTP의 기본 메서드를 살펴보겠습니다.

- GET / 리소스 조회 / 멱등성 O
- POST / 리소스 생성 / 멱등성 X
- PUT / 리소스 전체 수정 / 멱등성 O
- PATCH / 리소스 일부 수정 / 멱등성 X
- DELETE / 리소스 삭제 / 멱등성 O

이때 멱등성이란 같은 요청을 여러 번 보내도 결과가 동일한 성질입니다.

GET을 여러번 호출해도 서버 상태는 변하지 않지만, POST를 여러번 호출하면 리소스가 중복 생성될 수 있습니다.

---

### HTTP 상태 코드

- 2xx / 성공 / 200 OK, 201 Created
- 3xx / 리다이렉션 / 301 Moved Permanently, 302 Found
- 4xx / 클라이언트 오류 / 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found
- 5xx / 서버 오류 / 500 Internal Server Error

이때 401 에러와 403 에러가 헷갈릴 수 있는데

401 Authentication (인증) 은 누구인지 확인되지 않은 상태입니다. ( 로그인 필요한 경우)

403 Authrization (인가) 은 누구인지는 알지만 권한이 없는 상태입니다. ( 로그인은 했지만 접근 불가한 경우 )

### HTTPS와 TLS

HTTP는 데이터를 평문으로 전송하기 때문에 네트워크 상에서 데이터가 노출될 수 있습니다.

그래서 HTTP에 TLS(Transport Layer Security) 암호화를 적용해 전송 구간을 보호한 HTTPS를 사용하게 됩니다.

TLS Handshake는

- TCP 연결 수립
- 클라이언트가 지원하는 암호화 방식 전달
- 서버가 인증서 (공개키 포함) 전달
- 클라이언트가 인증서 검증
- 대칭키 생성 및 교환
- 이후 대칭키로 암호화 통신

의 순서로 이루어집니다.

이때 처음에는 비대칭키(공개키/개인키)로 안전하게 대칭키를 교환한 뒤, 실제 데이터 통신은 속도가 빠른 대칭키 암호화로 진행하게 됩니다.

### REST와 RESTful API

REST는 HTTP를 기반으로 자원(Resource)을 표현하고 상태를 전달하는 아키텍처 스타일입니다.

REST의 핵심 원칙입니다.

- 자원은 URI로 표현한다
- 행위는 HTTP 메서드로 표현한다
- 무상태 : 서버는 클라이언트 상태를 저장하지 않는다
- 일관된 인터페이스 : URI 설계, 응답 형식 등이 일관성을 가진다

---

### REST API 설계 시 고려사항

URI 설계 원칙

- 명사를 사용하고, 동사는 HTTP 메서드로 표현합니다.
- 소문자와 하이픈( - )을 사용합니다.
- 계층 관계는 슬래시로 표현합니다.
- 파일 확장자는 포함하지 않습니다.

---

### 쿠키와 세션

HTTP의 무상태 특성을 보완하는 방법으로 쿠키, 세션이 있습니다.

- 쿠키

쿠키는 클라이언트(브라우저)에 저장되며 서버 부하가 없지만, XSS/CSRF에 취약합니다.

- 세션

세션은 서버에 저장되며 보안성이 높지만, 서버 메모리를 사용하고 확장성이 제한됩니다.

### Spring 백엔드 요청 처리 흐름

Spring에서 실제 요청이 처리되는 전체 흐름을 살펴보겠습니다.

- 클라이언트 HTTP 요청

→ WAS (Tomcat) 수신

→ 서블릿 필터 체인 (인증/인가/로깅 등)

→ DispatcherServlet

→ HandlerMapping (어느 컨트롤러로 보낼지 결정)

→ Controller → Service → Repository

→ DB 응답

→ HTTP 응답 반환

Spring Security를 사용하면 필터체인 단계에서 JWT 검증이 이루어지고, `@Transactional` 은 AOP 프록시로 Service 계층에서 적용됩니다.
